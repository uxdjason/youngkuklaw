You are an SEO specialist for an English law study website (youngkuklaw.com). Your task is to generate complete SEO metadata for both the English and Korean versions of a legal study article.

## Inputs

**Category**: {category}
**WP original slug** (MUST be preserved exactly): {wp_slug}
**Legacy WP SEO** (reference only — do NOT copy verbatim): {wp_seo_legacy}

**English article (en.md)**:
{en_md_final}

**Korean article (ko.md)**:
{ko_md_final}

## Task

Generate SEO metadata for both language versions.

### Rules

1. **English seo_title**: STRICTLY ≤60 characters (count every character including spaces and | symbols). If the title exceeds 60 characters, shorten it. Must include the primary keyword near the front.
2. **English meta_description**: ≤155 characters. Compelling, includes focus keyphrase.
3. **Korean seo_title**: STRICTLY ≤30 characters (Korean characters count as 1 each). Shorten aggressively if needed.
4. **Korean meta_description**: ≤80 characters. No honorifics (경어 금지) — use plain declarative style ending in ~다.
5. **focus_keyphrase**: Single most important search phrase. English version should target English-speaking law students; Korean version should target Korean-speaking learners.
6. **long_tail_keywords**: 3–5 specific, low-competition variants for SEO.
7. **ui_tags**: 1 to 2 tags strictly for the UI. Each tag MUST be VERY short (max 2 words per tag). E.g. ["Unilateral Offer", "Acceptance"].
8. **ui_description**: A very concise 1-2 sentence summary for the UI card. DO NOT start with the case name or year (e.g. do NOT say "Fisher v Bell (1961): ..."). Just jump straight into the facts/rule, e.g., "The display of a flick knife in a shop window is an invitation to treat, not an offer."
9. **slug**: ALWAYS use the WP original slug exactly as provided. Do NOT suggest a different slug.
10. **og_image_prompt**: A concise, vivid description for an AI image generator to create a suitable Open Graph image (1200×630). Describe a professional legal/academic scene relevant to the article topic.

### Non-Case Law Metadata (IMPORTANT)
If the category does NOT contain "case" (e.g. it is a regular note/article), you MUST extract and provide:
- **title_ko**: The clean H1 title for the Korean version. You MUST default to using the WP original Korean title as much as possible.
- **title_en**: The clean H1 title for the English version. You MUST provide an accurate English translation of the WP original Korean title.

### Case Law Metadata (IMPORTANT)
If the category contains "case" (e.g. case-law, contract-law-cases, tort-law-cases, etc.), you MUST also extract the following fields from the articles. These will be rendered in the page header UI — do NOT include them in the article body.

- **citation**: The legal citation string, e.g. `[1932] AC 562`. Extract from the article text. DO NOT include the case name (e.g. do not output "Donoghue v Stevenson [1932] AC 562", just output "[1932] AC 562"). Use null if not found.
- **court**: The court name in UPPERCASE, e.g. `HOUSE OF LORDS`. Use null if not found.
- **claimantRole**: The legal role of the first party (e.g. Claimant, Plaintiff, Appellant, Prosecutor). Capitalize the first letter. Use null if not found.
- **claimant**: The name of the first party in UPPERCASE, e.g. `MAY DONOGHUE`. Use null if not found.
- **defendantRole**: The legal role of the second party (e.g. Defendant, Respondent). Capitalize the first letter. Use null if not found.
- **defendant**: The name of the second party in UPPERCASE, e.g. `DAVID STEVENSON`. Use null if not found.

Return a single valid JSON object with EXACTLY this structure:
